## 什么是 Prompt Engineering？

### 定义

**Prompt Engineering** 是设计和优化输入给 AI Agent 的提示词（Prompt），以引导其产生期望输出的技术和艺术。

### 为什么重要？

```
❌ 弱 Prompt:
[vibe-architect] 做个网站

可能的输出：
- 需要多轮对话澄清需求
- 方案不够具体
- 遗漏重要功能

---

✅ 强 Prompt:
[vibe-architect] 设计一个企业官网，要求：
1. 首页：产品展示、公司介绍、联系方式
2. 关于我们：公司历史、团队成员、企业文化
3. 产品中心：产品列表、详情页、分类筛选
4. 新闻资讯：行业新闻、公司动态
5. 客户支持：在线客服、FAQ、工单系统
技术栈：React + Next.js + TypeScript
设计风格：现代简约、蓝色主色调、响应式设计
SEO要求：语义化HTML、Meta标签、sitemap.xml

预期输出：
- 完整的页面结构
- 路由设计
- 组件划分
- 数据模型
```

## 核心原则

### 1. 清晰 (Clarity)

**原则**: 使用明确、无歧义的语言

```bash
# ❌ 模糊
[vibe-architect] 我想要一个比较好的API

# ✅ 清晰
[vibe-architect] 我需要一个RESTful API，满足：
- 符合OpenAPI 3.0规范
- 统一的错误处理
- JWT认证
- 请求速率限制（100 req/min）
- API版本控制（/api/v1/, /api/v2/）
```

---

### 2. 具体 (Specificity)

**原则**: 提供具体的细节和要求

```bash
# ❌ 泛泛而谈
[@vibe-coder] 写一个用户注册功能

# ✅ 具体详细
[@vibe-coder] 实现用户注册API端点：

**端点**: POST /api/auth/register

**请求体验证**:
- username: 3-30字符，字母数字下划线
- email: 有效邮箱格式
- password: 最少8字符，必须包含大小写字母和数字

**业务逻辑**:
1. 检查username和email是否已存在
2. 密码使用bcrypt加密（salt rounds: 10）
3. 创建用户记录（role: 'user', isActive: true）
4. 生成JWT token（7天过期）

**响应格式**:
成功 (201):
{
  "success": true,
  "message": "User registered successfully",
  "data": {
    "user": { "id", "username", "email" },
    "token": "jwt_token_here"
  }
}

失败 (400):
{
  "success": false,
  "message": "Validation failed",
  "errors": ["Email already exists"]
}
```

---

### 3. 结构化 (Structure)

**原则**: 使用结构化的格式组织 Prompt

```bash
# 推荐结构
[vibe-architect] 项目名称：[名称]

## 1. 项目背景
[描述]

## 2. 核心功能
### 2.1 功能模块A
- 需求1
- 需求2

### 2.2 功能模块B
- 需求1
- 需求2

## 3. 技术要求
- 前端：[技术]
- 后端：[技术]
- 数据库：[技术]

## 4. 非功能需求
- 性能：[要求]
- 安全：[要求]
- 可扩展性：[要求]

## 5. 交付物
请提供：
- [ ] 架构设计
- [ ] API文档
- [ ] 数据模型
```

---

### 4. 可操作 (Actionable)

**原则**: 提供清晰的可执行指令

```bash
# ❌ 模糊指令
[vibe-architect] 看看这个架构好不好

# ✅ 可操作指令
[vibe-architect] 审查以下架构设计，请：
1. 评估架构的可扩展性
2. 识别潜在的单点故障
3. 提出至少3个改进建议
4. 给出总体评分（1-10分）

架构内容：
[粘贴架构描述]
```

## 进阶技巧

### 1. Chain-of-Thought Prompting

引导 Agent 展示思考过程：

```bash
[vibe-architect] 设计一个微服务架构

请使用以下思考过程：

1. **需求分析**

   - 识别服务边界
   - 确定服务职责

2. **服务设计**
   - 设计每个服务的职责
   - 定义服务间通信
   - 设计数据存储

3. **架构设计**
   - 设计整体架构
   - 确定部署方式
   - 考虑可扩展性

4. **技术选型**
   - 选择技术栈
   - 说明选择理由

请逐步展示你的思考过程，并说明每个决策的理由。
```

### 2. Few-Shot Learning

提供示例让 Agent 学习：

```bash
[@vibe-coder] 实现用户认证功能

参考以下示例风格：

**示例 1: 密码加密**
```javascript
import bcrypt from 'bcrypt';

const hashPassword = async (password) => {
  const salt = await bcrypt.genSalt(10);
  return await bcrypt.hash(password, salt);
};
```

**示例 2: Token 生成**
```javascript
import jwt from 'jsonwebtoken';

const generateToken = (userId) => {
  return jwt.sign({ id: userId }, process.env.JWT_SECRET, {
    expiresIn: '7d'
  });
};
```

请按照类似的代码风格实现：
1. 用户注册
2. 用户登录
3. 密码验证
4. Token 生成和验证
```

## 🎯 实战技巧总结

### DO (推荐做法)

1. **提供充分的上下文**
   ```bash
   [vibe-architect] 设计一个任务管理系统
   背景：用于团队协作，10-50人团队
   功能：任务分配、进度跟踪、报告生成
   ```

2. **使用结构化格式**
   ```bash
   [vibe-architect] 项目：XXX
   ## 功能
   1. XXX
   2. XXX
   ## 技术栈
   - XXX
   - XXX
   ```

3. **明确输出要求**
   ```bash
   [vibe-architect] 请提供：
   1. 架构图
   2. 技术栈
   3. API文档
   ```

4. **提供示例**
   ```bash
   [@vibe-coder] 参考这个风格实现：
   [粘贴示例代码]
   ```

### DON'T (避免做法)

1. ❌ **上下文过少**
   ```bash
   [vibe-architect] 做个网站
   ```

2. ❌ **信息过载**
   ```bash
   [vibe-architect] [粘贴100页需求文档]
   ```

3. ❌ **自相矛盾**
   ```bash
   [vibe-architect] 使用React和Vue框架
   ```

4. ❌ **缺少约束**
   ```bash
   [@vibe-coder] 实现文件上传
   # 没说大小限制、文件类型、存储位置
   ```
